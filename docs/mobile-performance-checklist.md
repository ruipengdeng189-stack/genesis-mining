# 挖矿页移动端性能避坑清单

## 这次定位到的核心问题

1. **主交互按钮仍使用普通 `click`**
   - 支付按钮已经使用了更快的触摸响应绑定。
   - 挖矿中心按钮仍依赖普通 `click`，在实体手机上会出现触摸到实际业务执行之间的延迟感。
   - 结果：用户感觉“点下去了，但金币 / 粒子 / 反馈很久才出来”。

2. **热量轮询每 100ms 触发整页刷新**
   - 原来 `updateHeatAndWarnings()` 会每 `100ms` 调用一次 `updateDisplay()`。
   - `updateDisplay()` 不只是改数字，还会顺手刷新排行、任务、T4、研究室等面板。
   - 结果：移动端主线程长期被占用，点击主按钮时更容易掉帧。

3. **单次点击的特效过重**
   - 每次点击都可能触发：
     - Canvas 粒子
     - DOM 飘字
     - 按钮震动 / 发光
     - 暴击闪屏
     - DOM 爆散粒子
   - PC 可以扛住，但实体手机 GPU / 合成线程更容易被压满。

4. **音频链路对移动端过重**
   - 主点击音效本身是多振荡器叠加。
   - 背景音乐会持续创建更多音频节点。
   - 如果主按钮又先走 `pointerdown`，再走延迟 `click`，还可能让“声音先响、奖励后到”的割裂感更明显。

5. **移动端不应该把所有视觉规格照搬 PC**
   - 大量 `drop-shadow`、高亮、爆闪动画在手机上成本更高。
   - 同样一套发光参数，在 PC 上是“酷”，在手机上可能就是“卡”。

---

## 本次已落地的修复

1. **主按钮改成移动端快响应点击链路**
   - 挖矿按钮接入了 `bindResponsiveTap(...)`。
   - 单独设置更低的去重时间，避免影响连点手感。

2. **保留音频解锁，但去掉移动端重复音效感**
   - `pointerdown` 只负责解锁音频系统。
   - 实际点击音效在真正的挖矿动作时播放，减少“先响后到账”的延迟错觉。

3. **将热量轮询从“整页重绘”改成“局部 HUD 更新”**
   - `100ms` 定时器现在只更新热量 / 过热相关 UI。
   - 不再每 `100ms` 重算整页复杂面板。

4. **引入显示刷新合帧**
   - 高频数值刷新改为 `requestAnimationFrame` 合并执行。
   - 避免短时间内多次点击导致重复重绘堆叠。

5. **移动端粒子 / 爆散 / 动画降载**
   - 点击粒子数量减少。
   - 暴击爆散粒子数量减少。
   - 飘字存在时间缩短。
   - 移动端主按钮发光和爆闪动画改成更轻的版本。

6. **移动端背景音乐与点击音效轻量化**
   - 背景音乐节拍变稀疏。
   - 点击音效在移动端使用更轻的配置。
   - 减少移动设备 WebAudio 的实时负担。

## 后续做新游戏时的强制规则

### 1. 主玩法按钮必须优先支持移动端快响应
- 不要只绑普通 `click`。
- 核心玩法按钮优先走：
  - `pointerup`
  - `touchend`
  - 或统一封装的快速触摸绑定

### 2. 高频状态更新必须“局部刷新”
- 不要把温度、倒计时、能量条这种高频 UI 直接绑定到整页 `render`。
- 高频轮询只更新最小必要 DOM。

### 3. 主玩法点击链路不要同时堆太多反馈
- 一次点击最多保留：
  - 1 个主数值反馈
  - 1 组轻粒子
  - 1 个短音效
- 暴击 / 稀有掉落才允许升级成重反馈。

### 4. 排行 / 任务 / 商城不要跟随每次点击全量刷新
- 排行榜、任务面板、仓库、T4 系统应：
  - 按需刷新
  - 节流刷新
  - 面板打开时再全量刷新

### 5. 移动端特效预算必须单独配置
- 同一个玩法至少准备两套参数：
  - `desktopFx`
  - `mobileFx`
- 不要默认“PC 参数直接搬到手机”。

### 6. 背景音乐一定要有移动端低负载方案
- 移动端背景音乐要减少同时发声的音符数量。
- 点击音效要避免一次点击创建过多振荡器 / 节点。

---

## 推荐的移动端性能自检流程

1. 真机连续点击主按钮 20 秒  
   - 看是否出现明显掉帧、触控延迟、音画不同步。

2. 真机打开排行 / 背包 / 商城后再点主按钮  
   - 看多面板共存时是否卡顿。

3. 真机开启背景音乐后再测  
   - 看音频是否成为额外负担。

4. 真机在低电量模式 / 中端机型再测  
   - 不要只用 PC 浏览器模拟。

---

## 结论

这次问题的根因不是“网络慢”，也不是单一 CSS 问题，而是：

- **主交互事件选型不对**
- **高频整页刷新设计不合理**
- **移动端特效预算超标**
- **音频与视觉反馈没有按设备能力分级**

后续新游戏只要先按这份清单做，就能大幅减少“PC 正常、真机卡顿”的重复问题。
