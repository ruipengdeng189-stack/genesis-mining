# 游戏页移动端布局避坑清单

## 这次在跑酷页定位到的核心问题

1. **首屏主舞台占比过大**
   - 跑酷舞台原本长期使用接近 `46vh ~ 58vh` 的高度。
   - 在实体手机上，这会直接吃掉首屏大部分空间。
   - 结果：用户第一屏只能看到很少内容，必须频繁下滑才能看到任务、商店、赛季等核心信息。

2. **顶部信息卡过厚**
   - 标题区、说明文案、统计卡在移动端仍接近桌面版密度。
   - 结果：首屏视觉层级是完整了，但信息效率偏低，真正可操作区域被往下挤。

3. **移动端把“横向容量”误判成“不够放”**
   - 跑酷页之前把：
     - 顶部 3 个概览指标
     - 舞台 3 个 HUD 指标
     - 4 个方向按钮
     - 3 个动作按钮
   - 都压成了多行布局。
   - 结果：每个区块都多占一行，纵向空间被迅速拉长。

4. **页签内容跟整页抢滚动**
   - 如果整页一直自然向下铺，赛季 / 商店 / 任务这些长面板会把页面越拉越长。
   - 结果：用户频繁在“整页滚动”和“寻找功能”之间切换，体验很像网页，而不像 App。

5. **移动端需要“组件内滚动”，而不是一味整页滚动**
   - 长内容模块更适合放进限定高度的内容卡里，让内容区自己滚动。
   - 结果：可以保留入口和当前上下文，减少迷失感。

## 本次在跑酷页已落地的修复

1. **缩小首屏舞台高度**
   - 跑酷舞台改为基于 `dvh + clamp(...)` 的更小移动端高度。
   - 目标不是“塞满屏幕”，而是“舞台 + HUD + 操作 + 页签能同时尽量多露出一点”。

2. **压缩头部卡片密度**
   - 缩小顶部安全边距、卡片 padding、标题字号和说明行数。
   - 移动端说明文案会限制行数，优先保住标题、按钮和统计。

3. **把可横向排开的信息重新压回单行**
   - 顶部 3 个 summary 指标恢复 3 列。
   - 跑酷舞台 HUD 恢复 3 列。
   - 4 个方向按钮恢复 4 列。
   - 3 个动作按钮恢复 3 列。
   - 这样能明显减少纵向长度。

4. **把面板区改成“卡片内滚动”**
   - 页签区域在移动端改为限定高度。
   - 页签内容本身可以纵向滚动。
   - 这样用户切页签时不会被超长页面拖着跑。

5. **进一步压缩小屏手机样式**
   - 在 `420px` 以下进一步缩小：
     - 外边距
     - 卡片圆角
     - 标题字号
     - Canvas 高度
     - toast 与弹窗占位
   - 保证超小屏也不会只剩一个大舞台，其他内容全被挤下去。

6. **第二轮压缩长卡片密度**
   - 针对任务、商店、赛季、排行、说明弹窗进一步做了卡片级压缩：
     - 奖励胶囊更小
     - 卡片内段落限行
     - 长列表行高更紧凑
     - KPI 面板改成更适合手机的 2 列结构
   - 目标是让用户在不牺牲可读性的前提下，一屏内看到更多“真正可操作的信息”。

## 后续做新游戏时的强制布局规则

### 1. 先给首屏设“信息预算”
- 首屏至少要优先露出这 4 类内容中的 3 类：
  - 当前主玩法区
  - 即时关键数值
  - 当前核心操作按钮
  - 功能入口 / 页签
- 如果首屏只能看到 1~2 类内容，就说明布局过厚。

### 2. 移动端优先使用 `100dvh`
- 页面主容器、弹窗、浮层优先使用 `100dvh` 或 `calc(100dvh - safe-area)`。
- 不要只用传统 `100vh`，否则在部分手机浏览器会出现高度判断不准的问题。

### 3. 能横排的，不要轻易改成多行
- 如果一组内容只有 3~4 项，而且文案不长，优先尝试：
  - 缩小字号
  - 缩小 padding
  - 控制行数
- 不要第一时间改成两行或三行。

### 4. 长内容面板优先做“模块内滚动”
- 任务、背包、赛季、商城、排行等长内容：
  - 优先放进固定高度卡片
  - 卡片内 `overflow-y: auto`
- 不要让整页无限变长。

### 5. 移动端文案要有压缩版本
- 同一段描述至少准备：
  - 正常版
  - 移动端短版 / 限行版
- 尤其是首屏标题说明、卡片介绍、提示文案。

### 6. 小屏机型必须有二次压缩规则
- 建议至少有两个断点：
  - `max-width: 560px`
  - `max-width: 420px`
- 超小屏不应只是“整体等比缩小”，而是要继续削减非关键视觉占位。

### 7. 弹窗与 toast 也要算进移动端布局
- 不能只适配主页面。
- 弹窗、说明层、toast、底部按钮、页签条都要一起验证。

### 8. 长卡片优先做“信息分层压缩”
- 长卡片不要只靠整体缩放。
- 优先依次压缩：
  - 描述文案行数
  - 奖励胶囊尺寸
  - KPI 卡片列数
  - 行列表 padding / gap
- 这样比单纯缩小字号更不容易伤害可读性。

## 推荐的移动端布局自检流程

1. **首屏检查**
   - 真机打开页面，不滑动，确认是否能一眼看到：
     - 主玩法
     - 关键数值
     - 至少一组操作
     - 功能入口

2. **连续切页签检查**
   - 点开任务 / 商店 / 赛季 / 排行等入口。
   - 看是否需要长距离来回滚动才能回到主玩法。

3. **超小屏检查**
   - 用窄屏真机或浏览器模拟 `360 x 780` 左右尺寸。
   - 看标题、按钮、页签是否仍然完整可用。

4. **横竖向信息密度检查**
   - 如果一屏只能看到一个大模块，说明纵向浪费过多。
   - 如果一屏塞太多小元素且全都难点，说明横向压缩过头。

---

## 结论

这次跑酷页的问题，本质不是“功能太多”，而是：

- **移动端仍沿用了接近桌面的空间分配**
- **可横排的信息被错误改成了多行**
- **长内容区没有收敛到模块内滚动**

后续新游戏只要先按这份清单验一遍，就能大幅减少“手机上看起来内容很少、一直要往下翻”的重复问题。
